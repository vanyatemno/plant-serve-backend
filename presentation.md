## Слайд 2. Актуальність та наукова новизна

**Проблема:**
- Швидка та точна ідентифікація хвороб рослин є критичною для фермерів та любителів
- Існуючі рішення пропонують лише ймовірнісний клас хвороби без глибокого пояснення

**Наукова новизна:**
- **Гібридний підхід:** ML-класифікація + LLM-експертна система
- **Адаптивна поведінка:** різні стратегії при різних рівнях впевненості моделі
  - confidence < 0.75 → LLM як незалежний експерт-патолог
  - confidence ≥ 0.75 → LLM генерує конкретний план лікування
- Автоматичне генерування персоналізованих рекомендацій

### Текст виступу:
> Актуальність роботи зумовлена тим, що для багатьох користувачів — як фермерів, так і любителів кімнатних рослин — швидка та точна ідентифікація хвороб є критичною для збереження рослин. Існуючі рішення часто пропонують лише ймовірнісний клас хвороби без глибокого пояснення алгоритму дій.
>
> Наукова новизна нашого проєкту полягає у використанні гібридного підходу. Ми поєднали спеціалізовану модель комп'ютерного зору, натреновану на наборах даних хвороб рослин, із можливостями великих мовних моделей через OpenRouter API.
>
> Новизна розв'язку також в адаптивній поведінці системи: якщо ML-модель видає результат з низькою впевненістю (confidence менше 0.75), система не просто видає помилку, а передає топ-прогнози та контекст до LLM, яка виступає як незалежний експерт-патолог, аналізує всі варіанти і дає зважений висновок. У випадку високої впевненості — генерується конкретизований план лікування та превентивних заходів.

## Слайд 4. Архітектура застосунку

```
┌──────────────┐     ┌─────────────────────────────────────────────┐
│              │     │            NestJS Backend                    │
│   React      │────▶│                                             │
│   Client     │     │  ┌──────────┐  ┌────────────────────────┐  │
│              │◀────│  │ AuthModule│  │  PlantAnalysisModule   │  │
└──────────────┘     │  │ (JWT)    │  │                        │  │
                     │  └──────────┘  │  PlantAnalysisService  │  │
                     │                │    ├─ S3 Upload         │  │
                     │  ┌──────────┐  │    ├─ ML Service Call   │  │
                     │  │ Users    │  │    └─ AI Enrichment ────┼──┼──▶ OpenRouter API
                     │  │ Module   │  │                        │  │     (Anthropic Claude)
                     │  └──────────┘  │  AiEnrichmentService   │  │
                     │                │    └─ AIGeneratorService│  │
                     │                └────────────────────────┘  │
                     │                                             │
                     │  ┌──────────┐  ┌──────────┐               │
                     │  │ Prisma   │  │ Config   │               │
                     │  │ Module   │  │ Module   │               │
                     │  └────┬─────┘  └──────────┘               │
                     └───────┼─────────────────────────────────────┘
                             │
                     ┌───────▼─────┐     ┌──────────────┐
                     │ PostgreSQL  │     │  ML Micro-   │
                     │ Database    │     │  service     │
                     └─────────────┘     └──────────────┘
                                         ┌──────────────┐
                                         │   AWS S3     │
                                         └──────────────┘
```

**Потік обробки запиту:**
1. Клієнт → завантаження зображення
2. Backend → збереження в AWS S3
3. Backend → POST запит до ML мікросервісу з S3 ключем
4. ML сервіс → повертає prediction, confidence, plant_type
5. Backend → формує промпт та відправляє до AI Generator Service
6. AI → повертає структуровану пораду
7. Backend → зберігає результат у БД та повертає клієнту

### Текст виступу:
> Дозвольте детальніше зупинитися на архітектурі нашого застосунку. Вона побудована за принципом мікросервісної взаємодії та чіткого розділення відповідальності.
>
> Процес працює наступним чином: Клієнт на React відправляє зображення рослини на наш NestJS Backend. Backend завантажує фото у хмарне сховище AWS S3 та отримує унікальний ключ. Цей ключ передається до незалежного ML мікросервісу, який завантажує фото, проводить аналіз і повертає JSON із передбаченням — наприклад, "Полуниця з опіком листя" — та показником впевненості.
>
> На основі цих даних на бекенді вмикається AiEnrichmentService. В залежності від рівня впевненості моделі, ми формуємо динамічний промпт і направляємо його до LLM для генерації структурованої медичної поради.
>
> Фінальний результат, що включає первинний прогноз, оцінку здоров'я та план лікування від ШІ, зберігається в базі даних та повертається клієнту. Всі ці операції захищені JWT-авторизацією з механізмом ротації refresh-токенів.
>
> Важливо зазначити, що архітектура побудована на абстракціях: інтерфейс AIGeneratorService дозволяє легко замінити провайдера ШІ без зміни бізнес-логіки.

## Додаток: Можливі запитання та відповіді

**Q: Чому саме NestJS, а не Express напряму?**
> NestJS надає модульну архітектуру, вбудовану підтримку Dependency Injection, декораторів та TypeScript з коробки, що значно спрощує підтримку та масштабування проєкту.

**Q: Як забезпечується безпека даних користувачів?**
> Паролі хешуються через bcrypt, refresh-токени зберігаються як SHA-256 хеші в базі даних з механізмом ротації, а всі захищені ендпоінти перевіряють JWT через Passport.js стратегію.

**Q: Що відбувається, якщо AI-сервіс недоступний?**
> Система має graceful degradation — у випадку помилки API повертається повідомлення про тимчасову недоступність поради, але основний результат ML-аналізу все одно зберігається та повертається користувачу.
